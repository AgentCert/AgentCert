2 Methodology

Qualitative assessments are produced by a council of 3 independent LLM judges. Each judge evaluates every run on its own across reasoning quality, RAI compliance, and security posture. A meta-reconciliation step then combines individual assessments into a single consensus summary with severity labels and confidence ratings.

1
[Note] Evaluation lifecycle: Each run puts the agent through four phases — detection, diagnosis, remediation, and post-remediation verification — inside a controlled Kubernetes environment.
2
[Note] Numeric aggregation: All numbers come straight from telemetry — mean, median, P95, std dev, min, max for timing; success rates, false negative/positive rates, and token counts. No manual adjustments.
3
[Note] Qualitative aggregation: Three independent LLM judges evaluate each run separately. A meta-reconciliation step combines their outputs into consensus summaries with severity labels and confidence ratings.
4
[Note] Inter-judge agreement: Measured as the proportion of dimensions where all 3 judges gave the same severity label. An agreement of 1.0 means every judge agreed on every dimension for every run.
5
[Note] Safety propagation: A worst-case policy — if even one run in a category fails a safety check, the entire category is flagged. PII, hallucination, and malicious prompt detections are tracked per-run and rolled up to the category level.
6
[Note] Metrics collected: Timing (TTD, TTM), success rates (detection, mitigation, false negative, false positive), quality scores (reasoning, response quality, action correctness), safety (hallucination, RAI, security, PII), and resource usage (input/output tokens).
7
[Note] Fault taxonomy coverage: 3 of 5 canonical fault categories were tested — Application, Network, and Resource. Database/Storage and Security/Adversarial categories were not included in this run.
8
[Note] Aggregation pipeline: Individual run metrics roll up into fault-category scorecards, which then roll up into an overall agent scorecard. Numeric metrics use run-count-weighted means; qualitative assessments use LLM Council consensus.
9
[Note] Metrics not tracked: Recovery verification time (how long from mitigation to confirmed healthy state) is not broken out separately — it is included within TTM but not measured on its own.
10
[Note] Run count note: With small sample sizes, statistical measures like P95 should be treated as directional rather than definitive. The runs_per_fault field reflects the actual number of runs executed per fault type.

5 Detection & Response

Time-to-Detect (TTD) is how long it takes the agent to notice something is wrong. It is measured in seconds, starting from the moment a fault is injected until the agent raises its first detection signal.

Time-to-Detect (TTD) is how long it takes the agent to notice something is wrong. It is measured in seconds, starting from the moment a fault is injected until the agent raises its first detection signal.

Time-to-Mitigate (TTM) is how long it takes the agent to fully resolve a fault. It is measured in seconds, from fault injection to confirmed recovery. TTM includes detection time — it measures the entire journey, not just the fix. Because different runs can drive the median for TTD and TTM independently, median TTM can occasionally appear lower than median TTD.

Why do we show both Median and P95? The median gives you the typical result — it is not thrown off by a single slow run the way an average can be. P95 (the 95th percentile) tells you the worst-case you should plan for: 95% of runs finished within this time. Together they paint a fuller picture — one for typical behavior, one for tail risk. A note on small samples: with only 5 runs per category, the P95 is effectively the slowest run observed. These numbers are directional indicators, not statistically robust bounds.

Why can mitigation be 100% while detection is low? Detection rate tracks whether the agent explicitly identified the root cause before acting. Mitigation rate tracks whether the fault was resolved by the end of the run, regardless of how. In practice, the agent often fixes problems through its standard workflow — restarting unhealthy pods, cleaning up disk — without formally labeling the root cause. It acts on symptoms even when it does not flag the underlying fault. That is why you can see 100% mitigation alongside low detection.
