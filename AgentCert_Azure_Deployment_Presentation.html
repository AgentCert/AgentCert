Design notes

  • Core Litmus collections use snake_case field names such as project_id and experiment_id.
  • Newer agent and app registry collections use camelCase keys such as projectId, agentId, and appId.
  • Recent run snapshots are denormalized into chaosExperiments.recent_experiment_run_details for fast dashboard queries.
  • Detailed LLM traces and scoring observations are stored in Langfuse, while MongoDB keeps the platform state and run linkage keys.

Detailed Deployment & Runtime Flow

1. Build & PublishRepository scripts build GraphQL, auth, sidecar, LiteLLM, flash-agent, and install-agent images, then push them into Azure Container Registry.
2. Bootstrap ClusterNamespaces, RBAC, CRDs, service accounts, and Litmus controllers are established before rollout and restart logic is allowed to patch live workloads.
3. Deploy Control PlaneFrontend, auth, and GraphQL services are rolled out first so users, APIs, and registry workflows become available.
4. Activate Execution Planesubscriber, event-tracker, workflow-controller, chaos-operator, and chaos-exporter turn benchmark requests into Litmus workflow runs.
5. Run BenchmarkA benchmark definition links a target application, an agent, and a fault plan. The target is stressed, the agent reacts, and the platform records time-to-detect and time-to-recover.
6. Capture EvidenceOTEL spans, Langfuse traces, LiteLLM callbacks, Kubernetes logs, and Mongo-backed run state are correlated into a final verdict and audit trail.
Why this matters: The flow separates build-time concerns, cluster bootstrap, execution control, and evidence capture so failed deployments or controller issues can be isolated faster.

Operations, Scaling & Recovery

Scaling model
Scale frontend and API pods independently from experiment controllers. Use separate node pools if benchmark loads are materially heavier than control-plane traffic.
Recovery model
Preserve MongoDB state, Langfuse traces, Helm values, and exported benchmark evidence so failed runs can be replayed and audited after an outage.
Support model
Keep runbooks for namespace bootstrap, rollout failures, controller health, ACR pull issues, Key Vault access, and Langfuse or OTEL ingestion breaks.
  • Node pools: Consider dedicated pools for control plane, experiment controllers, and agent workloads when memory and CPU shapes differ.
  • Backups: Protect MongoDB state, exported workflow artifacts, and production Helm values on a defined retention schedule.
  • SLOs: Track API latency, rollout success, benchmark completion rate, controller restart frequency, and trace correlation quality.
  • Day-2 operations: Periodically validate namespace quotas, certificate expiry, ACR pull permissions, and workload image freshness.
